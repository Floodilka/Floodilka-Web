# 🎯 Best Practices для деплоя Boltushka

## 📊 Типы обновлений

### 1. 🟢 Минорные обновления (безопасные)
**Что:** Небольшие фиксы, улучшения UI, несрочные изменения

**Когда деплоить:**
- 🌙 Ночью (2-6 утра по МСК)
- 📉 В период минимальной активности
- 📅 В будние дни (избегать выходных если есть активность)

**Команда:**
```bash
sudo bash deployment/update-graceful.sh
```

**Downtime:** ~2-5 секунд

### 2. 🟡 Средние обновления
**Что:** Новые фичи, изменения API, обновления зависимостей

**Когда деплоить:**
- 🌙 Ночью или ранним утром
- ⚠️ Предупредить активных пользователей за 5-10 минут

**Процесс:**
```bash
# 1. Уведомить пользователей (если есть механизм)
bash deployment/notify-maintenance.sh 300  # 5 минут

# 2. Подождать
sleep 300

# 3. Обновить
sudo bash deployment/update-graceful.sh

# 4. Проверить
sudo -u boltushka pm2 logs --lines 50
```

**Downtime:** ~5-10 секунд

### 3. 🔴 Критичные обновления (hotfix)
**Что:** Исправление критических багов, security fixes

**Когда:** Немедленно

**Процесс:**
```bash
# Быстрое обновление
cd /var/www/boltushka
git pull
sudo bash deployment/update-graceful.sh
```

**Downtime:** ~2-5 секунд (неизбежно)

## 🛡️ Минимизация влияния на пользователей

### PM2 Reload vs Restart

**✅ Используйте `pm2 reload` (graceful):**
```bash
sudo -u boltushka pm2 reload boltushka-backend
```
- Новые запросы идут на новый процесс
- Старый процесс завершает текущие запросы
- Минимальный downtime

**❌ Избегайте `pm2 restart` (hard restart):**
```bash
sudo -u boltushka pm2 restart boltushka-backend  # только для отладки
```
- Жесткое убийство процесса
- Все соединения рвутся

### Что происходит с пользователями

#### ✅ При graceful reload:
- **HTTP API**: почти без разрыва
- **WebSocket**: ~2-5 сек разрыв, автопереподключение
- **Голосовые каналы**: разрыв P2P, нужно переподключиться
- **Текстовый чат**: сообщения не теряются (в памяти)

#### ❌ При restart:
- Все соединения рвутся
- Пользователи должны перезайти
- Возможна потеря несохраненных данных

## 📋 Чеклист перед деплоем

### 1. Подготовка
- [ ] Проверить изменения локально
- [ ] Убедиться что тесты проходят (если есть)
- [ ] Закоммитить и запушить в Git
- [ ] Выбрать время минимальной активности

### 2. На сервере (проверка)
```bash
# Проверить текущий статус
sudo -u boltushka pm2 status

# Посмотреть активность
sudo -u boltushka pm2 logs --lines 20

# Проверить место на диске
df -h

# Проверить память
free -h
```

### 3. Деплой
```bash
cd /var/www/boltushka
git pull
sudo bash deployment/update-graceful.sh
```

### 4. После деплоя
```bash
# Проверить что backend запустился
sudo -u boltushka pm2 status

# Смотреть логи в реальном времени 2-3 минуты
sudo -u boltushka pm2 logs boltushka-backend

# Проверить в браузере
# - Открыть сайт
# - Проверить основные функции
# - Зайти в голосовой канал
```

## 🚨 План отката (Rollback)

Если что-то пошло не так:

### Быстрый откат (Git)
```bash
cd /var/www/boltushka

# Посмотреть последние коммиты
git log --oneline -5

# Откатиться к предыдущему коммиту
git reset --hard HEAD~1  # или конкретный хеш

# Пересобрать
sudo bash deployment/update-graceful.sh
```

### Восстановление из бэкапа frontend
```bash
# update-graceful.sh создает бэкап в public.backup
cd /var/www/boltushka
rm -rf public
mv public.backup public
sudo systemctl reload nginx
```

### Восстановление backend
```bash
cd /var/www/boltushka/backend
git checkout HEAD~1 server.js  # откатить конкретный файл
sudo -u boltushka pm2 reload boltushka-backend
```

## 📊 Мониторинг после деплоя

### Первые 5 минут - критично
```bash
# В отдельном терминале смотреть логи
sudo -u boltushka pm2 logs boltushka-backend

# Проверить ошибки в nginx
sudo tail -f /var/log/nginx/boltushka-error.log
```

### Первый час - важно
- Проверить через 15, 30, 60 минут
- Убедиться что нет утечек памяти
- Проверить что пользователи подключаются

```bash
# Статистика PM2
sudo -u boltushka pm2 monit
```

### Первый день
- Периодически проверять логи
- Следить за отзывами пользователей
- Мониторить использование ресурсов

## 🎯 Рекомендации по времени деплоя

### Для русскоязычной аудитории (МСК)

**🟢 Оптимальное время:**
- 02:00 - 06:00 МСК (ночь, минимум пользователей)
- 14:00 - 15:00 МСК (обеденное время, но риск больше)

**🟡 Приемлемое время:**
- 09:00 - 11:00 МСК (утро, средняя активность)
- 22:00 - 00:00 МСК (поздний вечер)

**🔴 Избегать:**
- 18:00 - 22:00 МСК (пиковая активность)
- Выходные дни (если есть активность)
- Праздники

## 📊 Метрики для отслеживания

- **Uptime**: Цель >99.9%
- **Deployment downtime**: <10 секунд
- **Error rate**: после деплоя должен остаться на том же уровне
- **Response time**: не должно увеличиться после деплоя
- **Active connections**: восстановление в течение 1-2 минут
