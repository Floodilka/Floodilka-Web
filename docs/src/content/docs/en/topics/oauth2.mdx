Floodilka's OAuth2 implementation follows [RFC 6749](https://datatracker.ietf.org/doc/html/rfc6749). Use it to let Floodilka users log in to your site, or to request permission to act on their behalf.

## When to use OAuth2

- **"Sign in with Floodilka"** for a third-party site — request the `identify` scope.
- **Fetch the user's guild list** to show which of your servers they belong to — `identify guilds`.
- **Act on behalf of the user** (send messages as them, update their profile) — scope-specific.
- **Invite a bot to a server** — see [Bots](/topics/bots/). This uses OAuth2 but most of the flow is automatic.

## Endpoints

| Purpose | Method | URL |
|---------|--------|-----|
| Authorization page | `GET` | `https://floodilka.com/oauth2/authorize` |
| Token exchange | `POST` | `https://floodilka.com/api/v1/oauth2/token` |
| Token revocation | `POST` | `https://floodilka.com/api/v1/oauth2/token/revoke` |
| Current authorization info | `GET` | `https://floodilka.com/api/v1/oauth2/@me` |

## Scopes

| Scope | Description |
|-------|-------------|
| `identify` | Basic user info: username, discriminator, avatar, flags. |
| `email` | User's email address. Requires `identify`. |
| `guilds` | List of guilds the user is a member of. |
| `guilds.join` | Add the user to a guild you control (requires a prior relationship). |
| `bot` | Add a bot to a guild. See [Bots](/topics/bots/). |
| `connections` | Linked accounts (GitHub, Twitch, etc.). |
| `messages.read` | Read messages in channels the user can see. |

Ask for the **minimum** scopes you need. Users are more likely to authorize a modest request.

## Authorization code flow

The standard flow for server-side apps.

<Steps>

1. **Redirect the user** to the authorization page:

	```
	https://floodilka.com/oauth2/authorize
	  ?response_type=code
	  &client_id=YOUR_CLIENT_ID
	  &scope=identify%20email
	  &redirect_uri=https%3A%2F%2Fyour.app%2Fcallback
	  &state=RANDOM_CSRF_TOKEN
	  &prompt=consent
	```

2. **User approves.** Floodilka redirects to your `redirect_uri` with a `code` and the `state` you sent:

	```
	https://your.app/callback?code=AbC123&state=RANDOM_CSRF_TOKEN
	```

	Verify the `state` matches before continuing — this defends against CSRF.

3. **Exchange the code for an access token** on your server (never in the browser — this request uses the client secret):

	```bash
	curl https://floodilka.com/api/v1/oauth2/token \
	  -X POST \
	  -H "Content-Type: application/x-www-form-urlencoded" \
	  -d "grant_type=authorization_code" \
	  -d "code=AbC123" \
	  -d "redirect_uri=https://your.app/callback" \
	  -d "client_id=YOUR_CLIENT_ID" \
	  -d "client_secret=YOUR_CLIENT_SECRET"
	```

	Response:

	```json
	{
		"access_token": "...",
		"token_type": "Bearer",
		"expires_in": 604800,
		"refresh_token": "...",
		"scope": "identify email"
	}
	```

4. **Call the API** on the user's behalf:

	```bash
	curl https://floodilka.com/api/v1/users/@me \
	  -H "Authorization: Bearer ACCESS_TOKEN"
	```

</Steps>

## Refreshing an access token

Access tokens expire. Use the refresh token to get a new pair:

```bash
curl https://floodilka.com/api/v1/oauth2/token \
  -X POST \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=refresh_token" \
  -d "refresh_token=REFRESH_TOKEN" \
  -d "client_id=YOUR_CLIENT_ID" \
  -d "client_secret=YOUR_CLIENT_SECRET"
```

<Aside type="note">
	Refresh tokens are rotated on every use. Always store the `refresh_token` from the response — the old one stops working.
</Aside>

## Client credentials flow

For app-to-app requests that don't act on behalf of a specific user. Grants an access token scoped to the application itself.

```bash
curl https://floodilka.com/api/v1/oauth2/token \
  -X POST \
  -u "YOUR_CLIENT_ID:YOUR_CLIENT_SECRET" \
  -d "grant_type=client_credentials" \
  -d "scope=identify"
```

## Revoking tokens

When a user disconnects your integration, revoke their tokens so they can no longer be used:

```bash
curl https://floodilka.com/api/v1/oauth2/token/revoke \
  -X POST \
  -u "YOUR_CLIENT_ID:YOUR_CLIENT_SECRET" \
  -d "token=ACCESS_OR_REFRESH_TOKEN"
```

## Redirect URIs

Register every redirect URI in the application's **General** tab *exactly* as you will send it — including scheme, host, port, and path. Mismatches are rejected with `invalid_request`. For local dev, add `http://localhost:3000/callback` (or whatever port you use) alongside your production URI.
