OAuth2 в Floodilka следует [RFC 6749](https://datatracker.ietf.org/doc/html/rfc6749). Используйте его, чтобы дать пользователям Floodilka залогиниться на вашем сайте или запросить разрешение действовать от их имени.

## Когда использовать OAuth2

- **«Войти через Floodilka»** на стороннем сайте — скоуп `identify`.
- **Получить список гильдий пользователя** — чтобы показать, в каких ваших серверах он состоит — `identify guilds`.
- **Действовать от имени пользователя** (отправлять сообщения, менять профиль) — зависит от скоупа.
- **Добавить бота на сервер** — см. [Боты](/topics/bots/). Это тоже OAuth2, но большая часть потока автоматическая.

## Эндпоинты

| Назначение | Метод | URL |
|------------|-------|-----|
| Страница авторизации | `GET` | `https://floodilka.com/oauth2/authorize` |
| Обмен токенов | `POST` | `https://floodilka.com/api/v1/oauth2/token` |
| Отзыв токена | `POST` | `https://floodilka.com/api/v1/oauth2/token/revoke` |
| Информация о текущей авторизации | `GET` | `https://floodilka.com/api/v1/oauth2/@me` |

## Скоупы

| Скоуп | Что даёт |
|-------|----------|
| `identify` | Базовая информация: имя, дискриминатор, аватар, флаги. |
| `email` | Email пользователя. Требует `identify`. |
| `guilds` | Список гильдий, в которых состоит пользователь. |
| `guilds.join` | Добавить пользователя в вашу гильдию (нужна предварительная связь). |
| `bot` | Добавить бота в гильдию. См. [Боты](/topics/bots/). |
| `connections` | Привязанные аккаунты (GitHub, Twitch и т.д.). |
| `messages.read` | Читать сообщения в доступных пользователю каналах. |

Запрашивайте **минимум** нужных скоупов. Пользователи охотнее одобряют скромный запрос.

## Authorization code flow

Стандартный поток для серверных приложений.

<Steps>

1. **Перенаправьте пользователя** на страницу авторизации:

	```
	https://floodilka.com/oauth2/authorize
	  ?response_type=code
	  &client_id=ВАШ_CLIENT_ID
	  &scope=identify%20email
	  &redirect_uri=https%3A%2F%2Fyour.app%2Fcallback
	  &state=CSRF_TOKEN
	  &prompt=consent
	```

2. **Пользователь подтверждает.** Floodilka редиректит на ваш `redirect_uri` с `code` и вашим `state`:

	```
	https://your.app/callback?code=AbC123&state=CSRF_TOKEN
	```

	Убедитесь, что `state` совпадает, — это защита от CSRF.

3. **Обменяйте код на access token** на своём сервере (никогда в браузере — в запросе используется client secret):

	```bash
	curl https://floodilka.com/api/v1/oauth2/token \
	  -X POST \
	  -H "Content-Type: application/x-www-form-urlencoded" \
	  -d "grant_type=authorization_code" \
	  -d "code=AbC123" \
	  -d "redirect_uri=https://your.app/callback" \
	  -d "client_id=ВАШ_CLIENT_ID" \
	  -d "client_secret=ВАШ_CLIENT_SECRET"
	```

	Ответ:

	```json
	{
		"access_token": "...",
		"token_type": "Bearer",
		"expires_in": 604800,
		"refresh_token": "...",
		"scope": "identify email"
	}
	```

4. **Вызывайте API** от имени пользователя:

	```bash
	curl https://floodilka.com/api/v1/users/@me \
	  -H "Authorization: Bearer ACCESS_TOKEN"
	```

</Steps>

## Обновление access token

Access-токены истекают. Используйте refresh-токен, чтобы получить новую пару:

```bash
curl https://floodilka.com/api/v1/oauth2/token \
  -X POST \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=refresh_token" \
  -d "refresh_token=REFRESH_TOKEN" \
  -d "client_id=ВАШ_CLIENT_ID" \
  -d "client_secret=ВАШ_CLIENT_SECRET"
```

<Aside type="note">
	Refresh-токены ротируются при каждом использовании. Всегда сохраняйте `refresh_token` из ответа — старый перестаёт работать.
</Aside>

## Client credentials flow

Для app-to-app запросов, не связанных с конкретным пользователем. Выдаёт access-токен, привязанный к самому приложению.

```bash
curl https://floodilka.com/api/v1/oauth2/token \
  -X POST \
  -u "ВАШ_CLIENT_ID:ВАШ_CLIENT_SECRET" \
  -d "grant_type=client_credentials" \
  -d "scope=identify"
```

## Отзыв токенов

Когда пользователь отключает вашу интеграцию — отзывайте его токены, чтобы их нельзя было использовать:

```bash
curl https://floodilka.com/api/v1/oauth2/token/revoke \
  -X POST \
  -u "ВАШ_CLIENT_ID:ВАШ_CLIENT_SECRET" \
  -d "token=ACCESS_ИЛИ_REFRESH_TOKEN"
```

## Redirect URIs

Регистрируйте каждый redirect URI на вкладке **Общие** приложения *точно так же*, как будете отправлять — со схемой, хостом, портом и путём. Несовпадения отклоняются с `invalid_request`. Для локальной разработки добавьте `http://localhost:3000/callback` (или ваш порт) рядом с боевым URI.
